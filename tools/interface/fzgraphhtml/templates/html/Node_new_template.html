TD Property: unspecified [queue, Default]
Queued. TD is used only to indicate order in the queue. Apply this whenever other types do not strictly apply. Automatic rescheduling does take into account WORK UTD vs SELFWORK UTD.
Future: Ideally, automatic rescheduling of UTD Nodes should take into account the dependency structure.
 | inherit
Behaves just like the Superior Node it inherits from. ITD Nodes are those that are requirements for their superior Node(s) to be accomplished and where scheduling can be determined completely by the dependency hierarchy.
 | variable [movable after TD, Rare]
Automatically movable, but only if the TD has been passed. Assign the TD with that in mind. Try to apply this only to Nodes that are a priority that should be accomplished within 7 days as part of currently critical milestones. When applying this, carefully consider: Would this become a critical accomplishment with fixed TD in 7 days, 4 days, 2 days? Ask yourself: Is this crucial to keeping a bigger accomplishment on track that impacts many people?
These should be mainly about near-term objectives and Milestones. Having many VTD Nodes messes with the queued automatic rescheduling of UTD Nodes and is unrealistic.
 | fixed [real deadline, Relatively Rare]
Real deadline. The TD is so meaningful that the Schedule has to allow time to accomplish by that TD. Keep this very much in mind, resist the temptation to assign fixed TD where it does not truly apply, because doing so places a heavy burden on automatic scheduling. Ask yourself: Does this need to be completed by a specific time, because after that there are bad consequences or it becomes undoable, a missed opportunity, a broken promise? If the accomplishment requires significant time then the fixed TD should only be applied to the final component of a sub-hierarchy of Nodes.
FTD Nodes should be about things where the due date is to a large degree outside of my control and that are largely non-optional.
 | exact [real specific time, Externally Imposed]
Takes place at a real specific time. Normally, this is due to agreements with or expectations by other people. Ask yourself: Does this happen at a very specific time, like a meeting with others?
If an Exact TD Node requires preparation then that must be specified by a sub-hierarchy of Nodes. To those Nodes, apply either VTD or FTD type.
