System Documentation (v2.x)

by Randal A. Koene, 2024-

Preamble

This provides documentation about v2.x of the System, its aims, how to work with it, and about detail aspects of the System. Version 2.x of the System is associated directly with the v2.x implementation of the Formalizer. Formalizer tools will point to sections within this documentation.

DayWiz

Wiztable

Note that known Challenges and Decisions must be represented by at least one Node. There is a Trello Board about Challenges and Decisions.

Daywiz of previous day was finalized. Daywiz content for the previous day was updated and finalized as much as possible.
Review how I used my time (tool: dayreview). Use the dayreview.py Systemm tool to calculate a review of how time was allocated during the previous day. A performance score will be produced.

Note that an alternative visualization is offered by the new2023 sysmet categories automatic visualization.

Sync Google Calendar with Schedule. Look at the Google Calendar schedule and see if there are scheduled entries there that should have Node entry in today's schedule that do not yet have that. In those cases, make or find the appropriate Nodes and ensure they appear in the day's Schedule.
Review Promises. Update, revise and re-prioritize as needed. Make sure Scheduling is correct. Review the list of promise-Nodes in the Promises NNL.
  • Are there any that have been completed? If so, make sure they are moved to the promises_kept NNL.
  • Is the order of promises correct? Change target dates as needed.

Note that there is also a Trello board about Promises.

Confirm passed non-repeating Fixed Nodes to convert to Variable - or - manually update them. Use the method provided in the link to generate a NNL of passed non-repeating FTD Nodes. Then:
  • Manually update the target dates of those Nodes. Or,
  • Apply automated conversion from FTD to VDT type. This uses the NNL of passed non-repeating FTD Nodes in a fzserverpq API request, using a call implemented in fzupdate-cgi.py
    • .
Auto-update passed repeating Nodes to their next instances. Use the `both` or `repeating` options on the Incomplete Nodes page to auto-update repeating Nodes as needed.
Ensure that highest priority Promises are represented in the Schedule. Generate the Upcoming Incomplete Nodes page that highlights Nodes that are in the Promises NNL. Then:
  • Are promise-Nodes scheduled to be addressed soon?
  • Are there additional Nodes in the near-term Schedule that clearly do address promises but are not yet marked as such? If so, then add them to the Promises NNL.
Review progress on current Threads and their Scheduling. Ensure there are Nodes to complete upcoming Milestones. Review Threads and Milestones:
  1. Check the Threads Kanban to review that progress is being made and is sensible. This means, determine if the order of operations makes sense and make sure that work on Nodes is producing output that provides the prerequisites of other Nodes. Do the order and timing of Nodes in the Threads Kanban make sense?
  2. Check the Schedule with Threads highlighting to confirm that Threads-associated Nodes are predominant and that the proposed allocation of time to categories such as work and self-work is as desired.
  3. Check the Milestones Kanban to ensure that the arrangement of Milestones makes sense, that all the Milestones needed are there, and that Nodes exist to accomplish near-term Milestones. At present, this still needs a better visualization of the Milestones hierarchy for Milestones coming up!
  4. Create Nodes as needed, including Nodes that were suggested in the initial overview phase of the Schedule updating activity.

Note additional useful visualizations:

  • Active Main Categories
  • Milestone Categories
  • These are mapped to Milestones, Dependencies, Roadmap Path, Values and Urgency.
Additional notes:
  1. The "threads" Schedule marks only those Nodes that link up to Threads board top Nodes via their superiors hierarchy. Although the System, Sleep, Work Self-Work codes for other Nodes are not shown, those Boolean Flags do still apply to the daily review.
Identify Challenges and Decisions in the Schedule, add to NNL as needed.
  1. Check the Schedule to see if there are Nodes that should be marked as Challenges or challenging decisions that have not yet been marked as such. If so then add those Nodes to the NNL.
  2. Check the Challenges NNL to see if there are completed challenges that should be moved to the challenges_overcome NNL. These should also be reflected in the 'addressed at least 1 challenge' line of the previous day's wiztable.
Ensure schedule is mathematically doable. This means, check that the displayed sum of required times fits into the available time. Modify Node target dates as needed to achieve a fit.
Auto-update Variable Nodes. Use the 'variable' option (if 'both' was not already sufficient earlier) on the Incomplete Nodes page to auto-update VTD Nodes.

Note that this may be replaced with a different set of updates if undefined target date (UTD) Node type is repurposed as suggested.

Ensure Schedule is realistic. Review the Nodes in the Schedule and consider if the day's Nodes as suggested are a realistic plan.
  • To get better insight into this, you can review the Schedule as it would be shown in a Week Calendar Schedule.
  • Try to consider and account for mini-days in a bi-phasic sleep schedule.
  • Consider which types of Nodes are best addressed early in a (mini-)day or late in a (mini-)day.
Prepare the list of people that I already know I should communicate with this day. Add names to the Communications section of the DayWiz page. To find the names:
  • Consider names from the previous day where intended communication was not completed.
  • Review the Nodes Scheduled for the day and consider possible associated communications needed.
